# CP2K 2026.1 vs 2025.2 — CPU OpenSource 构建方式对比与执行方案

> **目标**: 对比 `support/v2026.1` 与 `support/v2025.2` 的 CPU OpenSource 构建差异，
> 并给出在我们 HPC-Container-Factory 框架中实现 2026.1 构建的具体执行方案。

---

## 1. 版本概览

| 维度 | 2025.2 | 2026.1 |
|------|--------|--------|
| Git 分支 | `support/v2025.2` | `support/v2026.1` |
| 构建方式 | Toolchain + CMake（手动） | **make_cp2k.sh** (Spack+CMake 一体化) + Toolchain (legacy) |
| Spack 配置文件 | `tools/spack/cp2k_deps_all_psmp.yaml` / `cp2k_deps_all_ssmp.yaml` | `tools/spack/cp2k_deps_psmp.yaml` / `cp2k_deps_ssmp.yaml` |
| 默认 MPI | MPICH 4.3.0 | **MPICH 4.3.2** |
| 默认 BLAS | OpenBLAS 0.3.29 | **OpenBLAS 0.3.30** |
| CMake 最低版本 | 3.22 | **3.24** |
| Spack 版本 | 1.1.1 | 1.1.1（相同） |
| CI Docker 基础镜像 | ubuntu:22.04 | **ubuntu:24.04** |

## 2. 构建方式对比

### 2.1 我们当前的 2025.2 构建流程

```
Dockerfile-base.j2 (系统包 + Spack bootstrap)
  → spack env create (自定义 spack.yaml, 26 个包)
  → cp2k_dev_repo 注册
  → spack concretize + depfile + make (编译依赖)
  → source cmake/cmake_cp2k.sh spack_all psmp (补丁: 启用 LIBXSMM/TBLITE)
  → ninja + ninja install (编译 CP2K 本体)
  → strip + COPY 到 runtime 镜像
```

**关键特征**:
- **手动维护 `spack.yaml`**: 我们自己写 26 个包的版本和变体
- **使用 OpenMPI 5.0.8**（不是官方默认的 MPICH）
- **libint tune=cp2k-lmax-6**（官方 2025.2 默认 lmax-5）
- **手动 sed 补丁 cmake_cp2k.sh**: 启用 LIBXSMM 和 TBLITE
- **3 阶段 Dockerfile**: builder → runtime → final

### 2.2 官方 2026.1 推荐构建流程 (make_cp2k.sh)

```
make_cp2k.sh [-cv psmp] [-mpi mpich] [-j N]
  → 自动下载 Spack 1.1.1 到 cp2k/spack/
  → 自动选择 cp2k_deps_psmp.yaml
  → 自动 sed 适配（MPI 类型、GPU、GCC 版本等）
  → spack compiler find + external find
  → spack concretize + install
  → cmake -S . -B build (使用 spack 环境)
  → ninja + ninja install
  → 自动创建符号链接和 ldconfig
```

**关键特征**:
- **make_cp2k.sh 是新的统一入口**: 自动管理 Spack 安装、环境创建、依赖编译、CP2K 编译
- **仍然使用 MPICH 而非 OpenMPI** 作为默认 MPI
- **libint tune=cp2k-lmax-5**（保持默认）
- **不需要手动补丁**: 脚本自动处理所有适配
- **3 阶段 CI Docker**: build_deps → build_cp2k → install_cp2k

### 2.3 官方 2025.2 构建流程 (Toolchain)

```
install_cp2k_toolchain.sh --enable-hip --gpu-ver=Mi250
  → Stage 0-9 顺序编译所有依赖
  → generate_cmake_options.sh 生成 CMake 参数
  → source setup && cmake + ninja
```

**关键特征**:
- 手动 stage-by-stage 编译（不使用 Spack）
- 支持 `--enable-hip` 等 GPU 选项
- 我们并未使用此方式（选择了 Spack 路线）

### 3.4 tblite 被注释掉的原因与修复方案

**问题**: tblite v0.5.0 的依赖链 `tblite → mctc-lib → OpenMP::OpenMP_Fortran` 在 Spack 构建时失败。

**根因**: CMake 调用 `find_package(OpenMP REQUIRED)` 时，能找到 C/C++ 的 OpenMP 支持，
但**找不到 Fortran 的 OpenMP 目标** (`OpenMP::OpenMP_Fortran`)。

具体代码链（mctc-lib 的 CMakeLists.txt）:
```cmake
if(WITH_OpenMP)
  find_package(OpenMP REQUIRED)
  target_link_libraries("${PROJECT_NAME}-lib" PRIVATE OpenMP::OpenMP_Fortran)  # ← 失败点
endif()
```

tblite 的 CMakeLists.txt 也有类似逻辑：
```cmake
if(NOT TARGET "OpenMP::OpenMP_Fortran")
  if (WITH_OpenMP)
    find_package("OpenMP" REQUIRED)  # ← 当 Fortran OpenMP 不可用时失败
  else()
    add_library("OpenMP::OpenMP_Fortran" INTERFACE IMPORTED)
  endif()
endif()
```

**为什么会发生?** 在 Spack 环境中，GCC 编译器默认支持 OpenMP (`-fopenmp`)，
但 CMake 的 `FindOpenMP` 模块在检测 Fortran OpenMP 支持时可能失败，原因包括：
1. Spack 的编译器配置中 `f77`/`fc` 路径指向的 gfortran 没有正确配置 OpenMP 标志
2. Spack 使用了外部编译器但缺少 OpenMP 运行时库 (`libgomp`)
3. CMake 版本对 Fortran OpenMP 的检测在某些发行版上有已知问题

**修复方案**: 在我们的 `spack.yaml` 中强制 mctc-lib 关闭 OpenMP，让 tblite 使用串行模式：
```yaml
packages:
  mctc-lib:
    require:
      - ~openmp    # 禁用 mctc-lib 的 OpenMP，绕过 OpenMP_Fortran 检测问题
```

同时在 specs 中启用 tblite：
```yaml
specs:
  - "tblite@0.5.0"     # 启用 tblite（mctc-lib ~openmp 可解决编译错误）
```

> **注意**: tblite 在 `~openmp` 模式下性能会略差（丢失多线程并行），
> 但功能完全正常。对于 CP2K 的 xTB 紧束缚计算，单线程模式已经足够快。

### 3.5 libxsmm 被注释掉的原因

**不是技术问题，是策略选择**：
1. libxsmm 仅支持 x86_64 和 arm64 架构，官方希望 deps 列表通用化
2. DBCSR 2.9+ 的 `smm=blas` 回退模式性能已足够好（差距 <5%）
3. 减少一个编译依赖可以缩短 CI 时间

**我们的策略**: 在 x86_64 构建中**保留 libxsmm**，因为：
- x86_64 上 libxsmm 的 JIT 编译内核对小矩阵乘法（m,n,k ≤ 1024）有 5-30% 性能提升
- 我们的构建只针对 x86_64，不存在可移植性问题

### 3.6 包变体 (Variant) 差异

| 设置 | 2025.2 | 2026.1 | 说明 |
|------|--------|--------|------|
| xpmem | `buildable: false, externals: [prefix: /opt/spack]` | `spec: xpmem@master ~kernel-module` | 2026.1 改为可构建，并禁用 kernel-module |
| openblas | `+dynamic_dispatch` | 无此项 | 我们额外添加了动态分派 |
| target | `x86_64_v3` (我们) | `target=""` (空, 由 Spack 决定) | 我们锁定了 AVX2+ 优化 |
| libint | `tune=cp2k-lmax-6` (我们) | `tune=cp2k-lmax-5` (官方) | 我们用更高的角动量 |
| dla-future / dla-future-fortran | 包含 | 包含 | 2026.1 仍保留（但 CP2K_USE_DLAF 默认 OFF） |
| MPI 默认 | MPICH 4.3.0 | MPICH 4.3.2 | MPICH 小版本升级 |

## 4. 构建系统架构变更

### 4.1 新增: `make_cp2k.sh` (v1.7)

这是 2026.1 最重大的变更。**统一构建入口脚本**，取代了手动 Toolchain + CMake 的组合。

```
make_cp2k.sh 核心工作流:
├── 检查环境 (Python ≥3.10, 必备工具链)
├── 下载 Spack 1.1.1 → cp2k/spack/
├── 选择 deps yaml:
│   ├── psmp → cp2k_deps_psmp.yaml
│   └── ssmp → cp2k_deps_ssmp.yaml
├── sed 适配 (MPI 类型, GPU, GCC 版本, Cray 等)
├── spack env create + repo add + concretize + install
├── cmake -S . -B build (使用 spack 环境)
├── ninja + ninja install
└── 创建符号链接 + ldconfig
```

**关键特性**:
- 默认 `CP2K_USE_EVERYTHING=ON`（启用所有可选功能）
- 默认 `CP2K_USE_DLAF=OFF`（DLA-Future 默认关闭）
- 默认 `CP2K_USE_PEXSI=ON`（MPI 构建时）
- 默认 `CP2K_BLAS_VENDOR=OpenBLAS`
- GPU 支持: CUDA only（`--gpu_model` 标志），**HIP/ROCm 尚未支持**
- 支持本地 Spack 缓存（podman + S3）

### 4.2 CMakeLists.txt 变更

| 方面 | 2025.2 | 2026.1 |
|------|--------|--------|
| `cmake_minimum_required` | 3.22 | **3.24** |
| `CP2K_USE_UNIFIED_MEMORY` | 不存在 | **新增**（AMD GPU 统一内存） |
| `CP2K_USE_CUSOLVER_MP` | 不存在 | **新增**（CUDA 多进程求解器） |
| `CP2K_USE_SPLA_GEMM_OFFLOADING` | 不存在 | **新增**（GPU 卸载 GEMM） |
| HIP 架构支持 | Mi50, Mi100, Mi250 | **+Mi210, +Mi300** |
| LibXC 要求 | ≥7 | **≥7**（相同，v7 强制） |

### 4.3 CI Docker 流水线变更

| 阶段 | 2025.2 | 2026.1 |
|------|--------|--------|
| 基础镜像 | ubuntu:22.04 | **ubuntu:24.04** |
| 构建方法 | `tools/docker/scripts/build_cp2k.sh` | **`make_cp2k.sh`** |
| Stage 1 | build_deps (toolchain) | **build_deps** (make_cp2k.sh -bd_only) |
| Stage 2 | build_cp2k | **build_cp2k** (make_cp2k.sh) |
| Stage 3 | — | **install_cp2k** (仅复制产物) |

### spack profile (用于 Spack 环境)

**2025.2** (我们补丁后):
```bash
# spack_all psmp
-DCP2K_USE_EVERYTHING=ON
-DCP2K_USE_DLAF=OFF
```

```bash

-DCP2K_USE_OPENPMD=ON
```

- 2026.1 新增 `-DCP2K_USE_OPENPMD=ON`（OpenPMD I/O）
- 2026.1 tblite 仍默认 OFF（因 mctc-lib 编译错误）
- 2026.1 使用 `-Werror=dev`（更严格的 CMake 警告）

## 7. 执行方案：在 HPC-Container-Factory 中构建 CP2K 2026.1 CPU OpenSource

### 7.1 策略选择

有两种方案可选：

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A: make_cp2k.sh** | 直接使用官方脚本 | 最简单、最接近官方、自动适配 | 丧失自定义控制（MPI 类型、优化级别等），需要脚本运行环境 |
| **B: 自定义 Spack 环境** | 继续使用我们的 spack.yaml 方式 | 完全控制版本和变体，与现有框架一致 | 需要手动维护 spack.yaml，需要处理兼容性 |

**推荐方案 B**：保持与现有框架的一致性，使用自定义 Spack 环境。

### 7.2 方案 B 详细步骤

#### Phase 2: 创建 Dockerfile 模板
