## 4. 文件传输：全量复制

### 4.1 从 Stage 2 打包

```bash
mkdir -p /tmp/stage3-transfer && cd /tmp/stage3-transfer

# 1. CP2K runtime (136M) — cp2k.psmp + libcp2k.so
podman exec cp2k-stage1 tar -cf - -C / opt/cp2k-runtime > cp2k-runtime.tar

# 2. Spack view (874M) — 所有 CPU 运行时库
#    ⚠️ 关键: 必须用 -h (dereference symlinks)，否则只打包 symlink 本身
#    Spack view 是 symlink farm，实际文件在 ./_view/ 隐藏目录
podman exec cp2k-stage1 tar -chf - -C / opt/hpc/view > spack-view.tar

# 3. Toolchain install (46M) — DBCSR 等 HIP 库
podman exec cp2k-stage1 tar -chf - -C / opt/cp2k-src/tools/toolchain/install > toolchain-install.tar

# 4. 缺失的 ROCm 计算库 (993M) — 仅 ldd 报告需要的
#    注: librocsolver.so 独占 828M
podman exec cp2k-stage1 bash -c '
TMPDIR=/tmp/rocm-missing && rm -rf $TMPDIR && mkdir -p $TMPDIR/lib
for lib in librocblas libhipblas libhipblaslt libhipfft librocfft librocsolver librocroller; do
    for f in /opt/rocm/lib/${lib}*; do
        [ -e "$f" ] && cp -a "$f" $TMPDIR/lib/
    done
done
tar -cf - -C /tmp/rocm-missing lib
' > rocm-missing-libs.tar

# 5. CP2K data files (67M) — 基组、势函数等
podman exec cp2k-stage1 tar -cf - -C / opt/cp2k-src/data > cp2k-data.tar
```

## 7. 关键经验 & 踩坑记录

### 7.1 Spack view 是 Symlink Farm — tar 必须用 -h

Spack 的 environment view 通过 symlink farm 指向 `opt/hpc/._view/<hash>/` 下的实际文件。
用普通 `tar -cf` 打包时只会打包 symlink 本身（12K 的文件），**不会**包含实际库文件。

**解决方案**: 使用 `tar -chf`（`-h` = dereference，将 symlink 替换为实际文件内容）。

```bash
# ❌ 错误 — 只打包 symlink
podman exec cp2k-stage1 tar -cf - -C / opt/hpc/view > spack-view.tar
# 结果: 12K

# ✅ 正确 — follow symlink 打包实际文件
podman exec cp2k-stage1 tar -chf - -C / opt/hpc/view > spack-view.tar
# 结果: 874M
```

### 7.2 非 complete ROCm 镜像缺少计算库

`rocm/dev-ubuntu-24:7.2.1`（非 complete）只包含 HIP runtime 核心（libamdhip64, libhiprtc 等）。
CP2K 运行时额外需要：rocblas, hipblas, hipblaslt, rocsolver, rocfft, hipfft, rocroller。

这些库总计 ~993M（其中 librocsolver 独占 828M），必须从 complete 镜像中补齐。

### 7.3 全量复制 vs 最小复制的取舍

| 策略 | 优点 | 缺点 |
|------|------|------|
| 最小复制 (仅 ldd 报告的 .so) | 镜像更小 | 可能遗漏 dlopen 加载的库，运行时报错风险高 |
| **全量复制** (整个 view + toolchain) | 运行时完整，不会缺库 | 镜像偏大（多 ~200-300M 开发头文件等） |

**决策**: 采用全量复制。磁盘便宜，运行时缺库的调试成本高。

### 7.4 `podman cp` vs `tar pipe` 的选择

`podman cp` 无法在两个容器之间直接复制，必须经过宿主机中转。对于大文件（GB 级别），`tar` pipe 方式更高效：

```bash
# ✅ 推荐: tar pipe — 无中间文件
cat spack-view.tar | podman exec -i cp2k-stage3 tar -xf - -C /

# ❌ 不推荐: podman cp — 需要宿主机临时空间
podman cp cp2k-stage1:/opt/hpc/view ./tmp/
podman cp ./tmp/view/ cp2k-stage3:/opt/hpc/
```

### 7.5 ⚠️ `podman commit` 不记录 bashrc 写入的 ENV

`podman commit` 只保存文件系统变更，**不会**将 `>> /etc/bash.bashrc` 写入的环境变量记录到 Docker image metadata。
Apptainer 转换 SIF 时只读取 `ENV` 字段，不看 bashrc。

**解决**: 必须通过 `--change 'ENV ...'` 注入：

```bash
# ❌ 错误 — bashrc 写入对 Apptainer 无效
podman exec cp2k-stage3 bash -c 'echo "export PATH=..." >> /etc/bash.bashrc'
podman commit cp2k-stage3 myimage  # ENV 不包含 PATH

# ✅ 正确 — 通过 --change 注入 Docker ENV
podman commit \
  --change 'ENV PATH=/opt/hpc/view/bin:/opt/cp2k-runtime/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin' \
  --change 'ENV CP2K_DATA_DIR=/opt/cp2k-src/data' \
  --change 'ENV ROCM_PATH=/opt/rocm' \
  --change 'ENV HIP_PLATFORM=amd' \
  --change 'ENV LD_LIBRARY_PATH=/opt/cp2k-runtime/lib:/opt/hpc/view/lib:/opt/rocm/lib' \
  cp2k-stage3 cp2k-stage3-runtime:latest
```

### 7.6 ⚠️ PATH 必须包含 `/opt/hpc/view/bin`（mpirun 所在位置）

mpich 的 `mpirun`、`mpiexec`、`hydra_pmi_proxy` 都在 `/opt/hpc/view/bin/`。
如果 PATH 不包含此路径，CP2K 的 `do_regtest.py` 会因为找不到 `mpirun` 而失败。

**症状**：
```
FileNotFoundError: [Errno 2] No such file or directory: '-n'
```
这是因为 `--mpiexec ""` 导致命令变成 ` -n 1 cp2k.psmp`，`-n` 被当作可执行文件名。

**解决**: PATH 必须包含 `/opt/hpc/view/bin`：
```bash
export PATH=/opt/hpc/view/bin:/opt/cp2k-runtime/bin:$PATH
export LD_LIBRARY_PATH=/opt/cp2k-runtime/lib:/opt/hpc/view/lib:/opt/rocm/lib
```

### 7.7 在 Apptainer 中运行 CP2K regtest

Apptainer 容器大部分目录只读，但测试只需要读取输入文件，输出写到可写目录。
使用 `python3 -u` 禁用输出缓冲以获得实时进度。

```bash

# 快速 smoke test（每个目录只跑第一个测试）
apptainer exec --rocm \
```

### 7.8 ⚠️ ROCm 计算库需要 `library/` 子目录（不仅仅是 .so）

**首次 regtest 结果**: 51/94 FAILED，全部因为 rocblas 找不到 TensileLibrary.dat。

```
rocBLAS error: Cannot read /opt/rocm/lib/rocblas/library/TensileLibrary.dat: Illegal seek for GPU arch : gfx942
rocBLAS error: Could not initialize Tensile host:
filesystem error: directory iterator cannot open directory: No such file or directory [/opt/rocm/lib/rocblas/library]
```

**根因**: 我们之前只复制了 `/opt/rocm/lib/*.so*` 文件，但 ROCm 计算库（rocblas、hipblaslt 等）还依赖 `library/` 子目录中的预编译 kernel 库（TensileLibrary）。

| 目录 | 大小 | 用途 |
|------|------|------|
| `rocblas/library/` | 644M | rocblas kernel 库（TensileLibrary.dat + .hsaco 文件），**必须** |
| `hipblaslt/library/` | 4.5G | hipblaslt kernel 库（可选，CP2K 未直接使用） |

**修复**: 必须额外复制 rocblas 的 library 目录：

```bash
# 从 Stage 2 复制 rocblas library 到 Stage 3
podman exec cp2k-stage1 bash -c 'tar -cf - -C /opt/rocm/lib rocblas/library' \
    | podman exec -i cp2k-stage3 tar -xf - -C /opt/rocm/lib

# 验证 gfx942 支持
podman exec cp2k-stage3 ls /opt/rocm/lib/rocblas/library/*gfx942*
```

### 7.9 ⚠️ `grid_unittest.psmp` 未被 ninja 默认构建

**首次 regtest 结果**: grid_unittest RUNTIME FAIL。

```
HYDU_create_process: execvp error on file /opt/cp2k-runtime/bin/grid_unittest.psmp (No such file or directory)
```

**根因**: `ninja` 默认不构建 `grid_unittest` 和 `grid_miniapp`，只构建 `cp2k.psmp` 和其他 unittest。必须手动指定目标。

**修复**: 在 Stage 2 中手动构建并复制：

```bash

    -o /opt/cp2k-runtime/lib/libcp2k.so.2026.1
cd /opt/cp2k-runtime/lib && ln -sf libcp2k.so.2026.1 libcp2k.so

# Copy 所有辅助二进制 (unittests, miniapps 等)
cp /opt/cp2k-src/build/bin/*.psmp /opt/cp2k-runtime/bin/
'
```

**预期结果**: `cp2k.psmp` (~59KB) + `libcp2k.so.2026.1` (~70MB) + 14 个辅助工具

## 9. GPU 依赖完整审计

### 9.1 CP2K 库使用架构：CPU + GPU 混合

CP2K 使用 **混合模式**：CPU 库来自 Spack view，GPU 库来自 ROCm。

| 库 | 来源 | 类型 | 用途 |
|---|---|---|---|
| OpenBLAS | `/opt/hpc/view/` | CPU | BLAS/LAPACK |
| FFTW3 | `/opt/hpc/view/` | CPU | FFT |
| ScaLAPACK | `/opt/hpc/view/` | CPU | 分布式线性代数 |
| ELPA (+ROCm) | `/opt/hpc/view/` | CPU+GPU | 特征值求解（含 ROCm GPU kernels） |
| libxc | `/opt/hpc/view/` | CPU | 交换相关泛函 |
| mpich | `/opt/hpc/view/` | CPU | MPI |
| **rocblas** | `/opt/rocm/lib/` | **GPU** | GPU BLAS |
| **rocsolver** | `/opt/rocm/lib/` | **GPU** | GPU LAPACK solver |
| **hipblas** | `/opt/rocm/lib/` | **GPU** | HIP BLAS wrapper |
| **hipblaslt** | `/opt/rocm/lib/` | **GPU** | HIP BLAS LT |
| **hipfft** | `/opt/rocm/lib/` | **GPU** | HIP FFT |
| **rocfft** | `/opt/rocm/lib/` | **GPU** | GPU FFT backend |
| **hiprtc** | `/opt/rocm/lib/` | **GPU** | HIP runtime compilation (JIT) |

> CP2K **不使用** AMD 的 CPU BLAS/LAPACK/FFTW。所有 CPU 端数学库来自 Spack (OpenBLAS/FFTW3/ScaLAPACK)。

### 9.2 GPU 库运行时数据依赖

ROCm 的 `.so` 文件之外，还需要 kernel library 目录：

| 目录 | 全部大小 | gfx942-only | 说明 |
|------|---------|-------------|------|
| `rocblas/library/` | 644M | **85M** (297 文件) | TensileLibrary.dat + .hsaco kernel |
| `hipblaslt/library/` | 4.5G | **275M** (1111 文件) | TensileLibrary contraction kernels |
| `rocfft/` | 1.7G | 不可拆分 | rocfft_kernel_cache.db (单一 SQLite DB) |
| **总计** | **6.9G** | **~2.1G** | |

**优化**: 只复制 gfx942 (Mi300X) 的 kernel 文件，节省 ~4.8G。

### 9.3 DBCSR HIP 运行时依赖

DBCSR `dbcsr-2.9.0-hip` 是静态链接（`.a`），HIP kernel 在编译时通过 `__hipRegisterFatBinary` 嵌入。
运行时需要：

| 依赖 | 用途 | Stage 3 状态 |
|------|------|-------------|
| `libamdhip64.so` | HIP runtime | ✅ 非complete 镜像自带 |
| `libhipblas.so` | GPU BLAS (dgemm) | ✅ 已复制 |
| `libhiprtc.so` | JIT 编译 (hiprtcCreateProgram 等) | ✅ 非complete 镜像自带 |
| LLVM bitcode (`*.bc`) | hiprtc JIT 编译的内置函数 | ✅ 非complete 镜像自带 (71 个 .bc) |
| `librocblas.so` + library/ | hipblas 底层调用 | ✅ 已复制 (gfx942-only 85M) |

DBCSR 使用 `hiprtc` 进行**运行时 JIT 编译** — 在运行时编译 HIP kernel 代码。
这需要 `/opt/rocm/lib/llvm/` 中的 LLVM bitcode（ocml.bc, ockl.bc, opencl.bc 等）。

### 9.4 ELPA ROCm 运行时依赖

ELPA `+rocm` 构建的 `libelpa_openmp.so.19` 链接了：

```
librocblas.so.5, librocsolver.so.0, libamdhip64.so.7,
libhipblaslt.so.1, librocprofiler-register.so.0,
libhsa-runtime64.so.1, libroctx64.so.4, librocroller.so.1
```

ELPA 的 GPU kernels 在编译时静态嵌入（非 JIT），不需要额外 kernel 数据文件。
但它通过 rocblas/rocsolver 执行 GPU 计算，所以 `rocblas/library/` 是必须的。

### 9.5 CP2K Grid HIP / DBM GPU 模块

CP2K 的 grid (collocation/integration) 和 dbm 模块有 HIP GPU 后端：

| 模块 | HIP 函数 | 额外数据依赖 |
|------|---------|-------------|
| grid/hip/ | `hipLaunchKernel`, `hipMemcpyToSymbol` | 无（编译时嵌入） |
| dbm/ | `hipEventCreate`, `hipFree`, `hipMalloc` | 无 |

这些模块的 HIP kernel 在编译时通过 `__hipRegisterFatBinary` 嵌入二进制，
不需要额外的 kernel 数据文件。

### 9.6 hipblaslt Sparse File 注意

`hipblaslt/library/` 中的 `.dat` 和 `.co` 文件使用了 **sparse file** 技术：
- `du -sh` 报告 3.0G（含 sparse hole）
- `du -sh --apparent-size` 也报 3.0G
- 实际磁盘占用约 275M

`tar` 默认保留 sparse file 属性，不会浪费空间。`podman save` 也正确处理。

### 9.7 最终 Stage 3 ROCm 库清单

| 路径 | 来源 | 说明 |
|------|------|------|
| `/opt/rocm/lib/*.so` | 非 complete 镜像 + 补充 | HIP runtime 核心 |
| `/opt/rocm/lib/rocblas/library/` | gfx942-only (85M) | rocblas TensileLibrary |
| `/opt/rocm/lib/hipblaslt/library/` | gfx942-only (275M) | hipblaslt TensileLibrary |
| `/opt/rocm/lib/rocfft/rocfft_kernel_cache.db` | 全量 (1.7G) | rocfft FFT kernel（不可拆分） |
| `/opt/rocm/lib/llvm/amdgcn/bitcode/` | 非 complete 镜像自带 | hiprtc JIT bitcode |
