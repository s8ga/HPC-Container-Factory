### 1.2 环境变量（⚠️ 关键 — 缺一不可）

```bash
# Spack view 必须在最前面（mpicc/mpirun 优先于系统）
export PATH=/opt/hpc/view/bin:$PATH
export PKG_CONFIG_PATH=/opt/hpc/view/lib/pkgconfig

# ⚠️ CPATH — toolchain 通过它找 ELPA 等头文件（缺少会导致 ELPA system detection 失败）
export CPATH=/opt/hpc/view/include:/opt/rocm/include

# ⚠️ LIBRARY_PATH — 链接时搜索（cmake find_library 使用）
export LIBRARY_PATH=/opt/hpc/view/lib:/opt/rocm/lib

# 运行时库搜索
export LD_LIBRARY_PATH=/opt/hpc/view/lib:/opt/rocm/lib:$LD_LIBRARY_PATH

# ⚠️ CMAKE_PREFIX_PATH — 必须同时包含 Spack view 和 ROCm
# 缺少 /opt/rocm 会导致 COSMA Tiled-MM 找不到 rocblasConfig.cmake
export CMAKE_PREFIX_PATH=/opt/hpc/view:/opt/rocm

# ROCm
export ROCM_PATH=/opt/rocm
export HIP_PLATFORM=amd
```

> **重要教训**：`CPATH`、`LIBRARY_PATH`、`CMAKE_PREFIX_PATH` 必须同时包含 `/opt/hpc/view` 和 `/opt/rocm`。
> 之前只设了 Spack view 没设 ROCm 路径，导致 COSMA HIP 构建找不到 `rocblasConfig.cmake`，
> ELPA system detection 找不到头文件。

## 2. Toolchain 脚本 Patch

### 2.1 ⚠️ 关键现实：CMake 支持 Mi300，但 toolchain `--gpu-ver` 不支持 Mi300

当前 `support/v2026.1` 的 CMake 侧支持 Mi300（映射 `gfx942`），但 toolchain 参数解析仍只接受到 `Mi250`。

**结论**：toolchain 阶段使用 `--gpu-ver=Mi250` 触发 HIP 构建；CMake 阶段再使用 `-DCP2K_WITH_GPU=Mi300` 指定目标架构。

> ⚠️ 只有 `get_openblas_arch.sh` 需要一个小 hack（见 §2.2），因为它尝试下载 OpenBLAS 来检测 CPU arch。

### 2.2 OpenBLAS Arch 检测 Hack（离线环境）

`scripts/stage0/get_openblas_arch.sh` 会从 `cp2k.org` 下载 OpenBLAS 来检测 CPU 架构。
在离线/网络受限环境中，替换为硬编码版本：

```bash
cat > /opt/cp2k-src/tools/toolchain/scripts/get_openblas_arch.sh << 'EOF'
#!/bin/bash -e
[ "${BASH_SOURCE[0]}" ] && SCRIPT_NAME="${BASH_SOURCE[0]}" || SCRIPT_NAME=$0
SCRIPT_DIR="$(cd "$(dirname "$SCRIPT_NAME")" && pwd -P)"
openblas_ver="0.3.30"
source "${SCRIPT_DIR}"/common_vars.sh
source "${SCRIPT_DIR}"/tool_kit.sh
source "${SCRIPT_DIR}"/signal_trap.sh
source "${INSTALLDIR}"/toolchain.conf
source "${INSTALLDIR}"/toolchain.env
! [ -d "${BUILDDIR}" ] && mkdir -p "${BUILDDIR}"
echo 'Skipping OpenBLAS arch detection (offline mode)'
echo 'OpenBLAS detected LIBCORE = ZEN (hardcoded)'
echo 'OpenBLAS detected ARCH    = x86_64 (hardcoded)'
cat << INNER > "${BUILDDIR}/openblas_arch"
export OPENBLAS_LIBCORE="ZEN"
export OPENBLAS_ARCH="x86_64"
INNER
EOF
chmod +x /opt/cp2k-src/tools/toolchain/scripts/get_openblas_arch.sh
```

> 如果网络可用，也可以跳过此 hack（toolchain 会自动下载检测）。

## 3. Toolchain 构建

### 3.1 获取 CP2K 源码


### 3.2 运行 Toolchain

```bash
cd /opt/cp2k-src/tools/toolchain

# 不需要 patch 脚本！但 toolchain 的 --gpu-ver 需用 Mi250（见 §2.1）

# 运行 toolchain
# 注意：不要用 --with-mpi=system（无效参数），MPI 通过 PATH 中的 mpiexec 自动检测
./install_cp2k_toolchain.sh \
    --with-cmake=system \
    --with-openblas=system \
    --with-fftw=system \
    --with-libxc=system \
    --with-libint=system \
    --with-scalapack=system \
    --with-elpa=system \
    --with-spglib=system \
    --with-libvori=system \
    --with-plumed=system \
    --with-dftd4=system \
    --with-tblite=system \
    --with-hdf5=system \
    --with-sirius=no \
    --with-cosma=no \
    --with-dbcsr \
    --enable-hip \
    --gpu-ver=Mi250 \
    -j$(nproc)
```
  使用 Mi250 的原因是 toolchain 只用于触发 HIP 依赖构建（这里是 DBCSR）。
  实际目标型号在 CMake 阶段通过 `CP2K_WITH_GPU=Mi300` 指定（对应 gfx942 / MI300X）。

> ⚠️ **注意**: `--with-mpi=system` 不是有效参数。toolchain 通过 `PATH` 中的 `mpiexec` 自动检测 MPI 类型。

### 3.4 Toolchain MPI 检测机制

Toolchain **不使用** `--with-mpi=system` 参数（该参数不存在）。MPI 检测逻辑：

```bash
# install_cp2k_toolchain.sh L340-365
if (command -v mpiexec > /dev/null 2>&1); then
  if (mpiexec --version 2>&1 | grep -s -q "HYDRA"); then
    export MPI_MODE="mpich"       # ← 我们的 Spack mpich 4.3.2 走这里
  elif ...
fi
```

只要 `PATH` 中有 `mpiexec`（Spack view 的 `bin/` 目录），toolchain 自动检测为 MPICH。

### 3.5 Toolchain dftd4/tblite 交互

当 `--with-tblite=system` 启用时，toolchain 自动将 dftd4 设为 `__DONTUSE__`：
```bash
# install_cp2k_toolchain.sh L835-836
if [ "${with_tblite}" != "__DONTUSE__" ]; then
  with_dftd4="__DONTUSE__"    # tblite includes dftd4
fi
```
这是正确的，因为 tblite 本身包含 dftd4 功能。

## 4. CP2K CMake 构建

### 4.1 ⚠️ 关键前提：完整环境变量 + toolchain setup

CMake 配置**必须**先设置关键环境变量，再 source toolchain setup，缺一不可：

```bash
cd /opt/cp2k-src
mkdir build && cd build

# 1. Spack view + ROCm 环境变量（关键变量，缺一不可）
export PATH=/opt/hpc/view/bin:$PATH
export PKG_CONFIG_PATH=/opt/hpc/view/lib/pkgconfig
export CPATH=/opt/hpc/view/include:/opt/rocm/include
export LIBRARY_PATH=/opt/hpc/view/lib:/opt/rocm/lib
export LD_LIBRARY_PATH=/opt/hpc/view/lib:/opt/rocm/lib:$LD_LIBRARY_PATH
export CMAKE_PREFIX_PATH=/opt/hpc/view:/opt/rocm
export ROCM_PATH=/opt/rocm

# 2. Toolchain setup（COSMA/DBCSR/libxsmm 路径 + 编译标志）
source /opt/cp2k-src/tools/toolchain/install/setup

# 3. setup 可能覆盖 pkg-config 搜索顺序，重新把 Spack view 放到最前
export PKG_CONFIG_PATH=/opt/hpc/view/lib/pkgconfig:${PKG_CONFIG_PATH:-}
```

> **仅 source setup 不够**！setup 脚本不会设置 `PATH`（MPI 在 view/bin），也不会设置
> `CMAKE_PREFIX_PATH`（cmake find_package 需要 `/opt/rocm` 找 rocblas 等）。
> 另外，`source setup` 后如果不重置 `PKG_CONFIG_PATH`，可能导致 CMake 阶段找不到
> `elpa_openmp.pc`。

### 4.2 ⚠️ DBCSR 版本号修复（必须，否则 cmake 失败）

Toolchain 构建的 DBCSR 版本字符串为 `2026.1.v2026.1v2026.1-dirty`（major=2026），
但 CP2K CMakeLists.txt 要求 `find_package(DBCSR 2.6)`（major=2），major 不匹配导致 cmake 失败。

**修复方法**：手动 patch 版本文件（两个：CPU 和 HIP）

```bash
# 将版本字符串从 "2026.1.v2026.1v2026.1-dirty" 改为 "2.9.0"
for f in /opt/cp2k-src/tools/toolchain/install/dbcsr-2.9.0/lib/cmake/dbcsr/DBCSRConfigVersion.cmake \
         /opt/cp2k-src/tools/toolchain/install/dbcsr-2.9.0-hip/lib/cmake/dbcsr/DBCSRConfigVersion.cmake; do
    sed -i 's/"2026.1.v2026.1v2026.1-dirty"/"2.9.0"/g' "$f"
done
```

> **根因**：DBCSR 源码在 CP2K repo 内部（`exts/dbcsr/`），toolchain 直接用 git describe 获取版本，
> 在 shallow clone 下得到的是 `2026.1.v2026.1v2026.1-dirty` 而非标准 semver。

#### ⚠️ 关键坑：ELPA pkg-config 可能被 toolchain setup 覆盖

即使 Stage 1 的 ELPA 已在 `/opt/hpc/view` 中，`source install/setup` 后 `PKG_CONFIG_PATH`
顺序被改变时，CMake 仍可能报找不到 `elpa_openmp`。解决方式是：

```bash
export PKG_CONFIG_PATH=/opt/hpc/view/lib/pkgconfig:${PKG_CONFIG_PATH:-}
```

#### ⚠️ 关键坑：HDF5 仅“找到版本号”不代表可用

实际遇到过 `Found HDF5 version ...` 但同时报 `missing HDF5_INCLUDE_DIRS`。稳定解法：

- Stage 1 使用 `hdf5+fortran+hl+mpi`
- CMake 显式传入并行 wrapper：`h5pcc` / `h5pfc`（见上方命令）

#### ⚠️ 关键坑：DFTD4/TBLITE 默认 OFF

`CP2K_USE_EVERYTHING` 默认为 `OFF`，所以 `CP2K_USE_DFTD4` 和 `CP2K_USE_TBLITE` 也默认 OFF。
**必须显式传 `-DCP2K_USE_DFTD4=ON -DCP2K_USE_TBLITE=ON`**，即使 toolchain 已检测到这些库。

同理，**FFTW3 也需要显式 `-DCP2K_USE_FFTW3=ON`**（Spack view 已有 fftw 3.3.10，库和 cmake config 均就位）。

#### ⚠️ `CP2K_ENABLE_ELPA_GPU` 已被弃用

CMake 会提示 "Manually-specified variables were not used: CP2K_ENABLE_ELPA_GPU"。
ELPA GPU 支持通过链接 `librocsolver.so` + `librocblas.so` 自动启用（Spack ELPA +rocm 变体自带）。

## 7. 关键经验总结

### Toolchain 发现 Spack 库的完整环境变量

```bash
export PATH=/opt/hpc/view/bin:$PATH
export PKG_CONFIG_PATH=/opt/hpc/view/lib/pkgconfig
export CPATH=/opt/hpc/view/include:/opt/rocm/include        # ← 头文件
export LD_LIBRARY_PATH=/opt/hpc/view/lib:/opt/rocm/lib:$LD_LIBRARY_PATH  # ← 运行时库
export CMAKE_PREFIX_PATH=/opt/hpc/view:/opt/rocm           # ← cmake find_package
export ROCM_PATH=/opt/rocm

# ⚠️ toolchain setup 设置 LIBRARY_PATH（含 libxsmm/dbcsr 路径），
# 必须用【追加】而非【覆盖】！否则 libxsmm/amdhip64 链接失败
source /opt/cp2k-src/tools/toolchain/install/setup
export LIBRARY_PATH=/opt/hpc/view/lib:/opt/rocm/lib:${LIBRARY_PATH}  # ← 追加到 setup 之后

# source toolchain setup 后，再次前置 PKG_CONFIG_PATH
export PKG_CONFIG_PATH=/opt/hpc/view/lib/pkgconfig:${PKG_CONFIG_PATH:-}
```

> **⚠️ LIBRARY_PATH 的时序至关重要**：
> 1. 先 `source setup`（设置 toolchain 的 libxsmm/dbcsr 路径）
> 2. 再**追加** Spack view 和 ROCm 路径
> 3. 如果先设 `LIBRARY_PATH` 再 `source setup`，setup 会**覆盖**你的设置
> 4. 如果手动 `export LIBRARY_PATH=...`（无 `${LIBRARY_PATH}`），会**丢失** setup 设置的路径

### Toolchain `with_cmake` 初始化 Bug 分析

`install_cp2k_toolchain.sh` 中 `with_cmake` **没有初始值**（不像 `with_ninja` 等默认 `__DONTUSE__`），
导致它在 bash 中是**空字符串 `""`**。

**依赖升级逻辑**（L841-855）：
```bash
# several packages require cmake.
if [ "${with_spglib}" = "__INSTALL__" ] || ... ; then
  [ "${with_cmake}" = "__DONTUSE__" ] && with_cmake="__INSTALL__"
fi
```
条件是 `${with_cmake} = "__DONTUSE__"`，但空字符串 `""` **不等于** `"__DONTUSE__"`，
所以依赖升级逻辑**不会触发**。`with_cmake` 依然是空字符串。

**后续 `install_cmake.sh` 的 case 分支**：
```bash
case "${with_cmake}" in
  __INSTALL__)  ...   # 不匹配
  __SYSTEM__)   ...   # 不匹配
  __DONTUSE__)  ...   # 不匹配
  *)                  # ← 空字符串走到这里，当作用户自定义路径处理
    pkg_install_dir="${with_cmake}"   # = ""
    check_dir "${with_cmake}/bin"     # check_dir "" → 报错
    ;;
esac
```

**结论**：不传 `--with-cmake` 时 toolchain 会因 `check_dir` 空路径报错。
这是 `install_cp2k_toolchain.sh` 的初始化遗漏——应该在默认值区域加上：
```bash
with_cmake="__DONTUSE__"   # 或 "__INSTALL__"
```

**解法**：始终显式传 `--with-cmake=system`（系统 cmake ≥ 3.28 即可）或 `--with-cmake=install`。

### HIP 构建的编译器架构（⚠️ 重要理解）

CP2K 的 HIP/ROCm 构建**不需要**将 C/C++/Fortran 编译器换成 AMD 编译器。
CMake 的 `enable_language(HIP)` 自动处理异构编译：

| 语言 | 编译器 | 用途 |
|------|--------|------|
| C | `/usr/bin/gcc` | Host C 代码 |
| C++ | `/usr/bin/g++` | Host C++ 代码 |
| Fortran | `/usr/bin/gfortran` | Host Fortran 代码 |
| **HIP** | `/opt/rocm/lib/llvm/bin/clang++` | **仅 GPU kernel** (`.cu.o` / `.hip.o`) |

这是因为 HIP 的工作模式是：**host 代码**（>99% 的 CP2K 源码）用 gcc/g++/gfortran 编译，
只有 GPU offload kernel（GRID collocation/integration、DBM multiply 等）通过 ROCm 的 clang++
编译为 `gfx942` 目标码。最终由 HIP linker (`clang++ --hip-link`) 将两者合并。

**验证方式**：检查 `build/CMakeCache.txt`：
```
CMAKE_C_COMPILER:FILEPATH=/usr/bin/gcc
CMAKE_CXX_COMPILER:FILEPATH=/usr/bin/g++
CMAKE_Fortran_COMPILER:FILEPATH=/usr/bin/gfortran
CMAKE_HIP_COMPILER:FILEPATH=/opt/rocm/lib/llvm/bin/clang++
```

| 分支/tag | Mi300 支持 | DBCSR 修复 | 推荐 |
|----------|-----------|------------|------|
| `v2026.1` (tag) | ❌ 需 3 个 patch | ❌ 硬编码 Mi250 | 不推荐 |

| `find_package(DBCSR 2.6) version mismatch` | DBCSR 版本 `2026.1...` | Step 4 的 sed patch |
| `elpa_openmp` not found by CMake | `PKG_CONFIG_PATH` 被 setup 覆盖 | source setup 后重新前置 PKG_CONFIG_PATH |
| `HDF5_INCLUDE_DIRS` missing | FindHDF5 并行+Fortran 场景不稳定 | 加 `-DHDF5_PREFER_PARALLEL=ON` |
| DFTD4/TBLITE/FFTW3 未启用 | `CP2K_USE_EVERYTHING` 默认 OFF | 显式传 `-DCP2K_USE_DFTD4=ON` 等 |
