# CP2K 2026.1 ROCm/HIP 混合构建方案（无 SIRIUS）

> 创建日期：2026-04-12
> 目标：为 AMD MI300X (gfx942) 构建 CP2K 2026.1 GPU 加速容器镜像
> 状态：**Phase 1 ✅ 已完成，Phase 2/3 实施中**

---

## TL;DR

为 AMD MI300X 构建 CP2K 2026.1 GPU 版本，采用 **混合构建**：

- ✅ **Spack 1.1.0** 安装 CPU + ROCm 加速依赖（12 顶层 specs，~78 包，其中 ELPA 启用 ROCm） — **已完成**
- 🔲 **CP2K toolchain** 仅构建 HIP 库（DBCSR + COSMA，仅 2 个）
- **不含 SIRIUS** → 不需要 SPLA/SpFFT 的 HIP 构建 → 5 个 patch 减少到 3 个
- **超越官方 CI**: ELPA GPU kernel 在官方 CI 中未启用 ROCm 构建，我们的方案提供更多 GPU 加速
- **编译器**: 全部使用 `%gcc@13.3.0`（非 rocmcc），ELPA HIP 代码通过 Spack ROCm build system 自动使用 hipcc
- **ELPA 额外 patch**: 需自定义 cassert patch 修复上游 bug（`elpa_cholesky_hip.hip` 缺少 `#include <cassert>`）

## 用户决策

- ✅ **不含 SIRIUS** — 跳过平面波，简化 HIP 构建
- ✅ **使用 mpich** — 跟随官方 ROCm CI
- ✅ **混合构建模式** — Spack(CPU) + Toolchain(HIP libs)
- ✅ **保留其余功能** — COSMA, PLUMED, DFTD4, TBLITE, ELPA, LIBXC, LIBINT 等

## 与 CI 构建对比

| 项目 | CP2K CI (Mi50/Mi100) | 我们的方案 (Mi300) |
|------|----------------------|--------------------|
| HIP 库 | DBCSR + COSMA(?) | DBCSR + COSMA |
| 额外库 | 仅 LIBXC + LIBINT2 | +PLUMED, DFTD4, TBLITE, ELPA |
| 构建方式 | 纯 toolchain | Spack(CPU+ROCm) + toolchain(HIP) |
| ELPA GPU | ❌ 未启用（toolchain 仅 NVIDIA kernels） | ✅ `+rocm` AMD GPU kernels |
| DLA-Future | ❌ 显式关闭 (`CP2K_USE_DLAF=OFF`) | ❌ 跳过（umpire/blt 不兼容 ROCm 7.x） |
| GPU | Mi50, Mi100 | **Mi300** (需 patch) |
| hipblas | apt install（CI 显式安装） | ROCm 7.2.1-complete 自带 |
| CI 覆盖 | ✅ 官方测试 | ⚠️ 无 CI 覆盖，需自行验证 |

## Phase 2: Toolchain 脚本 Patch（仅 3 个）

跳过 SIRIUS 后，SPLA/SpFFT 的 HIP 构建不再需要，patch 从 5 个减至 3 个。

### Patch 1: `install_cp2k_toolchain.sh` — `--gpu-ver` 验证

- **位置**: L684-709 case 语句
- **改动**: 添加 `| Mi300` 到有效 GPU 型号值列表
- **帮助文本同步更新**
- **影响**: 允许 `--gpu-ver=Mi300` 参数通过验证

### Patch 2: `install_cp2k_toolchain.sh` — `ARCH_NUM` 映射

- **位置**: L1143-1182 case 语句
- **改动**: 在 `Mi250)` 分支后添加 `Mi300)` 分支
- **影响**: 为 Mi300 设置正确的 GPU 架构编号

### Patch 3: `install_dbcsr.sh` — 关键 bug 修复 ⚠️

- **文件**: `scripts/stage9/install_dbcsr.sh` L74
- **改动**: `-DWITH_GPU=Mi250` → `-DWITH_GPU=${GPUVER}`
- **这是上游 bug**: DBCSR 硬编码 Mi250 而非使用 `--gpu-ver` 参数
- **影响**: DBCSR HIP 构建使用正确的 GPU 型号

### COSMA 不需要 Patch ✅

COSMA 的 HIP 构建逻辑只检查 `ENABLE_HIP` + `librocblas` 是否存在（无 GPU 型号 case 语句），使用 `-DCOSMA_BLAS=ROCM` 即可，天然兼容 Mi300。

- ~~`install_spla.sh`~~ — SPLA 不再需要 HIP 构建
- ~~`install_spfft.sh`~~ — SpFFT 不再需要 HIP 构建

    -DCP2K_WITH_GPU=Mi300 \
    -DCP2K_USE_MPI=ON \
    -DCP2K_USE_LIBXC=ON \
    -DCP2K_USE_LIBINT2=ON \
    -DCP2K_USE_COSMA=ON \

    -DCP2K_USE_SIRIUS=OFF \
    -DCP2K_USE_DLAF=OFF \
    -DCP2K_ENABLE_ELPA_GPU=ON \
    -DCP2K_USE_PEXSI=OFF \
    -DCP2K_USE_LIBSMEAGOL=OFF \
    -DCP2K_USE_DEEPMD=OFF \
    -DCP2K_USE_ACE=OFF \
    -DCP2K_USE_TREXIO=OFF \
    -DCP2K_USE_GREENX=OFF \
    -DCP2K_USE_LIBTORCH=OFF \
    -DCP2K_USE_MIMIC=OFF \

### 3.3 关键环境变量

```dockerfile
# ROCm 环境
ENV ROCM_PATH=/opt/rocm
ENV HIP_PLATFORM=amd
ENV PATH=$PATH:$ROCM_PATH/bin
ENV LD_LIBRARY_PATH=$LD_LIBRARY_PATH:$ROCM_PATH/lib

# Spack CPU libs → Toolchain 发现机制
ENV CMAKE_PREFIX_PATH=/opt/hpc/view
ENV PKG_CONFIG_PATH=/opt/hpc/view/lib/pkgconfig
ENV LD_LIBRARY_PATH=/opt/hpc/view/lib:$LD_LIBRARY_PATH
```

### 3.4 Toolchain 发现 Spack 库的机制

`--with-*=system` 时，toolchain 通过以下路径查找：

| 路径 | 机制 | 用途 |
|------|------|------|
| `CMAKE_PREFIX_PATH` | CMake `find_package()` | 库/头文件定位 |
| `PKG_CONFIG_PATH` | `pkg-config --modversion` | 版本检测 |
| `LD_LIBRARY_PATH` | 搜索 `.so` 文件 | 动态链接 |
| `CPATH` | 头文件搜索 | 编译时头文件 |
| `LIBRARY_PATH` | 搜索 `.a`/`.so` | 链接时库 |

Spack environment view 天然提供 `lib/`, `include/`, `lib/pkgconfig/` 结构，与 toolchain 的发现机制完全兼容。

### 3.5 Toolchain `--with-*=system` 与 Spack view 的集成

实际构建中，Spack 安装的 mpich@4.3.2 也包含在 view 中。因此 toolchain 调用应使用：

```bash
./install_cp2k_toolchain.sh \
    --with-openblas=system \
    --with-fftw=system \
    --with-libxc=system \
    --with-libint=system \
    --with-scalapack=system \
    --with-elpa=system \
    --with-mpi=system \
    --with-spglib=system \
    --with-libvori=system \
    --with-plumed=system \
    --with-dftd4=system \
    --with-tblite=system \
    --with-sirius=no \
    --with-cosma=install \
    --with-dbcsr \
    --enable-hip \
    --gpu-ver=Mi300
```

> ⚠️ 注意 `--with-mpi=system` 而非 `--with-mpich=system`。toolchain 的 MPI 检测
> 通过 `mpirun`/`mpicc` 等 wrapper 来发现，需要 Spack view 的 `bin/` 在 PATH 中。

## CI 对齐说明

基于对 CP2K 官方 CI (`tools/docker/`, `cmake/`) 的详细分析，我们的方案与官方 ROCm CI 的关键差异：

| 方面 | 官方 CI 做法 | 我们的做法 | 理由 |
|------|-------------|-----------|------|
| 构建工具 | 纯 toolchain | Spack + toolchain | Spack 管理更多依赖 |
| ELPA GPU | ❌ 不启用（toolchain 仅 NVIDIA kernels） | ✅ `+rocm` AMD GPU | Spack 有原生 AMD GPU 支持 |
| DLA-Future | ❌ `CP2K_USE_DLAF=OFF` | ❌ 同样 `OFF` | umpire/blt@0.4.1 不兼容 ROCm 7.x |
| hipblas | apt install `hipblas` | ROCm 7.2.1-complete 自带 | 无需额外安装 |
| MPICH LTO fix | `sed -i 's/-flto=auto//g'` | 同样需要 | 避免 LTO 与 HIP 编译器冲突 |
| HIP flags | `-munsafe-fp-atomics` (Mi250X+) | 同样需要 | CMakeLists.txt 自动添加 |
| GPU arch | `gfx906`(Mi50), `gfx908`(Mi100) | `gfx942`(Mi300) | CMakeLists.txt 已定义映射 |
| 测试 GPU | Mi50, Mi100 | Mi300 | 无官方 CI 覆盖 |

## 后续扩展路径

验证基础版后，可逐步添加：

- **Level 2a**: +SIRIUS（平面波支持，需额外 2 个 patch: SPLA/SpFFT Mi300 case）
- **Level 2b**: ~~DLA-Future~~ — 已移除（umpire/blt@0.4.1 不兼容 ROCm 7.x，CI 也未使用）
- **Level 3**: ✅ 已纳入主方案 — ELPA GPU kernel（`+rocm` 启用 `ELPA_2STAGE_REAL_AMD_GPU`）
- **Level 4**: +SPLA GPU offloading（`CP2K_USE_SPLA_GEMM_OFFLOADING=ON`，CI 对 GPU 构建默认开启）

### ~~Level 2b: DLA-Future~~（已移除 ❌）

DLA-Future 原计划通过 Spack `+rocm +scalapack` 构建，但在 Stage 1 实践中发现 umpire 依赖的
blt@0.4.1 其 `FindHIP.cmake` 在 ROCm 7.x 下失败（搜索 `$ROCM_PATH/hip/bin/hipconfig`，
但 ROCm 7.x 已改为 `/opt/rocm/bin/hipconfig`）。CP2K 官方 CI 的 ROCm 构建也显式设置
`CP2K_USE_DLAF=OFF`，因此我们决定移除 DLA-Future，与官方 CI 保持一致。

ELPA +rocm 仍然保留，提供 GPU 加速的特征值求解（`--enable-amd-gpu`）。
